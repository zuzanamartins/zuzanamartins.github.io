Category Three:
Databases

This is the continuation of the full stack project I introduced in Category Two - Algorithms and Data Structures. This second part focuses on my skills with Databases.

Identification of the artifact

The artifact that I chose for the third category, Databases, was created for the class CS-340 Client/Server Development. The class was taken in February of 2021. The course material introduced full stack development. The artifact that I chose is used to cover two categories: Algorithms/Data Structures and Databases, and for the Databases enhancement category, the back end of the project will be the focus. As previously mentioned, this project is a combination of two parts: the MongoDB database that is the topic of this category, and a Jupyter Notebook dashboard that acts as the user interface to the database. The user interface displays what are in fact MongoDB query results connected to the database through a CRUD (create, read, update, delete) class. The dashboard then takes those results and displays them through a variety of widgets. The Mongo queries allow for very specific and multifaceted search queries to be shown on a very user-friendly interface.
I selected this project not only because it is a full-stack artifact, and thus allows me to use a wide range of skills, but also because I found it quite challenging to both create, and further enhance. It gave me the chance to build something from the ground up and see my database come alive through a customizable user interface that even someone with no computer science background could easily use.

Overview of original functionality

The artifact itself is made of three files, a python CRUD (create, read, update, delete) class, Jupyter Notebook file and a comma-separated values text file The third category focuses on databases; thus the focus in this enhancement plan is on the MongoDB database, and in no small part also the Jupyter notebook file. The MongoDB database processes the comma-separated value into its own database and collection. To provide some basic search functions for functionality testing, I created a couple of simple index queries. The MongoDB database also holds the authentication data. It contains a separate collection that holds a user authentication; a full access admin account that is password protected. The Jupyter notebook contains a few blocks of Mongo queries meant to return specific results to the data table. However, in the original project file, they do not function.

Justification of performed enhancements

As part of my enhancement plan, I created fully working environments for the MongoDB, Python, and Jupyter Notebook. I imported the file containing database data and created the actual database and two collections: for the animal entries and for authentications. In the authentication collection, I created another user, one with read-only permissions to enable viewing of the database through the dashboard without risking unwanted changes to any data. Either authentication is required to access the database through both the terminal, and Jupyter Notebook. I corrected the Mongo queries within the notebook file, as those were previously incorrect and did not return any results to the data table when a radio button with pre-determined filter criteria was selected.

Explanation of enhanced functionality

With modifications and enhancements completed, the project successfully runs. The dashboard has hard-coded authentication that is verified against the database before access is allowed. From there, the data table is loaded with animal data entries that user can further manipulate by either manually sorting and filtering, or by using one of the four radio buttons that were created to filter out very specific types of animals; the queries check for specific breeds, age ranges, outcome types and more. Those same table results are then also displayed through the chart and map located below. It automatically adjusts to the first row, unless the user selects a different one.
